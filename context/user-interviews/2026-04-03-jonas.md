## Key Insights from Jonas

### What he liked
- **Easy deployment**: user doesn't care how it's deployed, it just works
- **All apps in one place**: solves the problem of code scattered across folders (Cursor creates thousands of folders, hard to find latest version)
- **Step-by-step setup**: no need to read docs for integrations (e.g. API keys), just follow the questionnaire
- **Not a lovable**: doesn't force a specific frontend or limit features, not trying to be Cursor either

### His framing of the product (3 core features)
1. **Easy deployment** - apps just work, no AWS setup needed
2. **All projects in one space** - structured dashboard for all agents/automations
3. **Access rights and permissions** - control agent development within a company

### Additional ideas he suggested
- **Sandbox/playground vs production**: split test version from live version, critical so you're not editing production. Name it simply ("test version" / "live version") for non-technical users
- **IP ownership**: companies want to own the agents employees create, don't want employees taking them when they leave
- **Enterprise permissions**: org-level environment settings controlling what permissions all generated apps have

### Strategic advice
- **De-prioritize marketplace** for now - focus on the three core features first, marketplace can come later
- **Agent-first vs hybrid**: referenced YC podcast about Supabase winning because of best documentation for agents. If going agent-first, need best docs. But current demo is more consumer-facing. Hybrid approach is fine.
- **Current reality**: people still need UI to trust agents, even if the future is agents
- Consulting firms are a wedge but this applies to **every company** - all worried about employees using tools with too many permissions
